Docker for Web Applications

A two-lesson classroom walkthrough where we use Docker to solve a problem we already recognize: getting the same web application to run the same way on everyone’s machine.

Lesson 1: Images + Containers Lesson 2: Docker Compose + PostgreSQL Walkthrough Project-Based

How We Are Going to Approach Docker

Before we start typing commands, I want us to agree on the purpose of today’s topic. Docker is not here to make our application more complicated. Docker is here because web applications have a lot of moving parts: Python, packages, environment variables, databases, ports, and sometimes different operating systems.

We have all seen some version of this problem: the project works on one computer, but it fails on another. One person has the right Python version. Someone else forgot to create the virtual environment. Someone has PostgreSQL installed, someone else does not. Docker gives us a way to describe the environment clearly so the application has a better chance of running consistently.

Main idea: Docker is not replacing our application. Docker packages our application together with the environment it needs.

Our App Base

We will use a familiar backend application, such as RecipeShare or another Flask app we already understand.

Our Timeline

We will spend two lessons on this. In the first lesson we run containers and containerize the app. In the second lesson we add a database with Docker Compose.

Our Goal

By the end, we want to start the application and database together with one command: docker compose up.

What We Should Be Able to Do by the End

By the end of these two lessons, we should be able to explain and use Docker in a practical way. I do not expect us to know every Docker command. I do expect us to understand what problem Docker solves and how to use it for a normal web application.

  • Explain the difference between a Dockerfile, an image, and a container.
  • Run a container that already exists on Docker Hub.
  • Build our own Docker image for a Flask application.
  • Run a web app container and open it in the browser.
  • Explain why the database should usually be a separate service.
  • Use Docker Compose to run the web application and PostgreSQL together.
  • Use environment variables to configure an app inside containers.
  • Use volumes so database data does not disappear every time we stop a container.
  • Troubleshoot common Docker errors without panicking.

The Mental Model I Want You to Keep

If Docker feels confusing at first, it is usually because three words get mixed together: Dockerfile, image, and container. Let’s use a kitchen analogy.

Say this out loud: We write a Dockerfile. Docker builds an image. Docker runs a container from that image.
Lesson 1

Running Containers and Containerizing Our App

Today I want us to get the first “magic moment.” First we will run something that someone else already packaged. Then we will package and run our own application.

Part A — Why Are We Doing This?

Let’s imagine we are about to share our final project with another developer. What do they need before the app can run?

  • They need the correct version of Python.
  • They need the correct packages installed.
  • They may need database software installed.
  • They need the right environment variables.
  • They need to know which command starts the app.

That is a lot of instructions, and every instruction is a place where something can go wrong. Docker lets us capture many of those instructions in files that can live in the project repository.

Important: Docker reduces environment problems. It does not remove the need to understand routes, models, migrations, databases, or application code.

Part B — Check That Docker Is Available

Let’s start by checking whether Docker is installed and whether Docker Compose is available.

docker --version

docker compose version
Mac and Windows setup

Docker Desktop should be installed before class. On Windows, Docker Desktop usually uses WSL 2. If the command fails, open Docker Desktop and wait until it says the Docker engine is running. Our MCON-364 VMs already have Docker Desktop setup

Part C — Run Our First Container

Now we will run Docker’s simplest test container.

docker run hello-world

Let’s slow down and describe what happened:

  • Docker checked whether the hello-world image was already on our computer.
  • If it was not there, Docker downloaded it from Docker Hub.
  • Docker created a container from that image.
  • The container ran, printed a message, and stopped.

Notice something important: not every container keeps running forever. Some containers do one job and exit. A web server container keeps running because it is waiting for requests.

Part D — Run a Web Server Container

Now let’s run a real web server. We are not writing the web server ourselves. We are using an existing image called nginx.

docker run -p 8080:80 nginx

Open this in the browser:

http://localhost:8080
Port mapping: Nginx listens on port 80 inside the container. Our computer is using port 8080. The flag -p 8080:80 connects those two ports.

When we are done, stop the container with Ctrl + C.

Part E — Look Around

Now that we have run a few containers, let’s inspect what Docker knows about them.

docker ps

docker ps -a

docker images
Command What I Want You to Notice
docker ps This shows containers that are currently running.
docker ps -a This shows all containers, including containers that already stopped.
docker images This shows images that are available locally on our computer.

Part F — Make Sure Our Flask App Works Without Docker

Before we Dockerize anything, we should prove that the app works normally. Docker should not be used to hide a broken application.

python -m venv .venv
source .venv/bin/activate      # Mac/Linux
# .venv\Scripts\activate      # Windows
pip install -r requirements.txt
flask run

If this project starts with python run.py, use that instead. The exact command depends on the application. The goal is simply to make sure the app runs before we add Docker.

Part G — Add a .dockerignore File

Next, we will tell Docker which local files should not be copied into the image. This is similar in spirit to .gitignore.

.venv
__pycache__
*.pyc
.git
.gitignore
.env
instance
.pytest_cache
.DS_Store
Think about it this way: The Docker image should contain what the app needs to run. It should not contain our virtual environment, Git history, cache files, or local secrets.

Part H — Create the Dockerfile

Now we are ready to write the recipe. Create a file named Dockerfile in the project root.

FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

EXPOSE 5000

CMD ["flask", "run", "--host=0.0.0.0", "--port=5000"]

Let’s read it line by line

Line How We Should Read It
FROM python:3.12-slim Start with an existing Python image. We do not build Linux and Python from scratch.
WORKDIR /app Inside the container, use /app as the project folder.
COPY requirements.txt . Copy the dependency list first.
RUN pip install ... Install the Python packages inside the image.
COPY . . Copy the rest of our application code.
EXPOSE 5000 Document that this app expects to listen on port 5000.
CMD ... Tell Docker the default command to run when the container starts.
Important detail: We use --host=0.0.0.0 because the app must listen for traffic from outside the container. If Flask only listens on 127.0.0.1 inside the container, the browser on our computer may not be able to reach it.

Part I — Build the Image

Now we ask Docker to build an image from the Dockerfile.

docker build -t recipeshare .

The dot at the end matters. It means “use the current folder as the build context.” In plain English, Docker is allowed to look at the files in this project folder while building the image.

Part J — Run Our App Container

Now we run a container from our own image.

docker run -p 5000:5000 recipeshare

Open:

http://localhost:5000

At this point, the Flask server may start, but the application may still fail when it tries to use the database. That is expected for many real Flask apps. The container has Python and our application code, but it still needs the correct database connection settings.

Important Docker idea: Inside a container, localhost means “inside this container.” If PostgreSQL is running directly on our laptop, the Flask container cannot reach it by using localhost. From inside the container, we use host.docker.internal to point back to the host computer.

Use Two Environment Files

For this first Docker demo, PostgreSQL is still running locally on our computer. That means we need one database URL for running Flask normally and a different database URL for running Flask inside Docker.

Create two files in the project root:

.env.local
.env.docker

The regular local file uses localhost, because Flask is running directly on our computer:

# .env.local
FLASK_APP=app
FLASK_ENV=development
DATABASE_URL=postgresql://postgres:postgres@localhost:5432/recipeshare

The Docker version uses host.docker.internal, because Flask is running inside a container and PostgreSQL is running outside the container on our computer:

# .env.docker
FLASK_APP=app
FLASK_ENV=development
DATABASE_URL=postgresql://postgres:postgres@host.docker.internal:5432/recipeshare
Read the two URLs carefully: The username, password, port, and database name are the same. The only part that changes is the host: localhost for normal local Flask, and host.docker.internal for Flask running inside Docker.

Copy the Docker Environment File into the Image

For this classroom demo, we can copy the Docker-specific environment file into the container as .env. That lets the application keep looking for a normal .env file.

Update the Dockerfile so the Docker environment file is copied into the image after the application code is copied:

FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .
COPY .env.docker .env

EXPOSE 5000

CMD ["flask", "run", "--host=0.0.0.0", "--port=5000"]
Security note: This is okay for a classroom local demo with fake local credentials. In a real deployed application, we usually do not bake secrets into the image. We pass environment variables at runtime or use a secret manager.

Rebuild and Run Again

Because we changed the Dockerfile and environment file, rebuild the image:

docker build -t recipeshare .

Then run the container again:

docker run -p 5000:5000 recipeshare

On Mac and Windows with Docker Desktop, host.docker.internal usually works automatically. On Linux, we may need to add this option:

docker run --add-host=host.docker.internal:host-gateway -p 5000:5000 recipeshare

Now the Flask app is still containerized, but it is connecting to the PostgreSQL database running on our own computer. In the next lesson, we will improve this setup by putting PostgreSQL into its own Docker container and using Docker Compose.

Cleaner alternative: pass the environment file when the container starts

Instead of copying .env.docker into the image, we can pass it at runtime:

docker run --env-file .env.docker -p 5000:5000 recipeshare

This is closer to how we usually want to handle environment-specific configuration, because the same image can be reused with different settings.

Lesson 2

Docker Compose, PostgreSQL, and Multiple Services

Last time, we successfully ran a Flask app in a container. That is a big step, but it is not the whole story. Real web applications usually need more than one thing running. The app needs a database. Sometimes it needs a frontend. Sometimes it needs Redis, a worker, or another service.

Part A — Quick Recap

Let’s remind ourselves of the flow:

Dockerfile
instructions
→
Image
built app package
→
Container
running app

Today we add one more idea: instead of starting each container manually, we describe the whole local application environment in one file.

Part B — Why Docker Compose?

Imagine starting the app by hand every day. We would have to remember the command for the web app, the command for the database, the ports, the passwords, the database name, and the order to start everything.

Docker Compose lets us write those choices once.

docker compose up

With one command, Docker Compose can start:

  • our Flask application
  • PostgreSQL
  • a private Docker network so the services can talk to each other
  • a volume so the database data can survive restarts

Part C — Environment Variables

Our app should not hard-code the database connection. Instead, it should read configuration from environment variables. For example:

DATABASE_URL=postgresql://postgres:postgres@db:5432/recipeshare
Very common mistake: Inside the web container, localhost means “this same web container.” It does not mean the database container. In Docker Compose, the database hostname is usually the service name, such as db.

Part D — Create docker-compose.yml

Now create a file named docker-compose.yml in the project root.

services:
  web:
    build: .
    ports:
      - "5000:5000"
    environment:
      FLASK_APP: app
      FLASK_ENV: development
      DATABASE_URL: postgresql://postgres:postgres@db:5432/recipeshare
    depends_on:
      - db

  db:
    image: postgres:16
    environment:
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
      POSTGRES_DB: recipeshare
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:

Let’s read the Compose file slowly

Part What It Means
web This is our Flask application service.
build: . Build the web image using the Dockerfile in this folder.
db This is our PostgreSQL service.
depends_on Start the database container before starting the web container.
volumes Store database files outside the temporary container layer.
Notice: The database container uses a ready-made PostgreSQL image. Our web service uses our own Dockerfile because that is where our application code lives.

Part E — Start the Full Application

Now we start both services together.

docker compose up --build

Open:

http://localhost:5000

If we see errors, we do not guess randomly. We read the logs and ask: is this a Python problem, a database problem, a port problem, or a migration problem?

Part F — Run Commands Inside the Web Container

Once the app runs in Docker, application commands should also run inside Docker. That way, we are using the same Python packages and environment variables that the container uses.

docker compose exec web flask --version

If the app uses Flask-Migrate, run migrations inside the web container:

docker compose exec web flask db upgrade

If we changed the models and need to create a new migration:

docker compose exec web flask db migrate -m "add recipe reviews"
docker compose exec web flask db upgrade
Key idea: The container is now the application environment. When we run app commands inside the container, the command sees the same dependencies and environment variables as the running app.

Part G — Compose Commands We Will Actually Use

Command When We Use It

Part H — Exercise: Prove the Database Persists

Let’s prove that the volume matters.

  1. Start the app with docker compose up.
  2. Create a recipe, user, or other record.
  3. Stop the app with Ctrl + C.
  4. Run docker compose down.
  5. Start again with docker compose up.
  6. Check whether the data is still there.

If the data is still there, that is because the named volume postgres_data preserved the database files.

Then restart:

Common Errors We Should Expect

When Docker fails, I do not want us to treat it like a mystery. Most beginner Docker errors fall into a few categories.

Port is already allocated

This means another process is already using the port on your computer.

Error response from daemon: Ports are not available: bind: address already in use

Two reasonable fixes:

  • Stop the process that is using the port.
  • Use a different host port, such as 5050:5000.
Flask runs, but the browser cannot connect

The app may be listening only on 127.0.0.1 inside the container. For Docker, use:

flask run --host=0.0.0.0 --port=5000
Database connection refused

Check the database hostname. Inside Docker Compose, use the service name:

postgresql://postgres:postgres@db:5432/recipeshare

Do not use localhost from inside the web container to reach the database container.

Changed code, but Docker still shows the old behavior

The image may need to be rebuilt.

docker compose up --build
The database connects, but tables are missing

The database may be running, but the schema has not been created yet. Run migrations:

docker compose exec web flask db upgrade

Adding the Next.js Frontend to Docker Compose

Our RecipeShare project has two separate repositories:

  • recipeshare — the Flask backend. This is where docker-compose.yml lives.
  • recipeshare-ui / recipes-frontend — the Next.js frontend. This has its own Dockerfile and is built into a Docker image separately, before Compose is started.

The reason we separate these concerns is practical: the frontend is a standalone deployable artifact. We build it once into a named image, and then Compose simply pulls that image by name — just like it pulls postgres:16 from Docker Hub. The backend team does not need the frontend source code in their folder.

The architecture we are aiming for is:

frontend
Next.js · port 3000
↔
web
Flask API · port 5000
↔
db
PostgreSQL · port 5432
Recommended order:
  1. Get the Flask backend and PostgreSQL working with Compose.
  2. Build the frontend image with docker build.
  3. Add the frontend service to docker-compose.yml referencing the pre-built image.
  4. Bring everything up with docker compose up.

Part A — How the Frontend Calls the Backend

All API calls in the frontend go through a thin helper file, lib/api.js. It reads a single environment variable — NEXT_PUBLIC_BACKEND_URL — to know the base URL. When that variable is not set (local dev), the base URL is empty, which means requests go to the same host the browser is already on (e.g. http://localhost:3000/api/recipes). Next.js then rewrites that path to the real Flask URL behind the scenes.

// lib/api.js  (excerpt)

// Empty string in dev (requests go to the same origin, Next.js rewrites them).
// In production you could set this to a fully-qualified URL if needed.
const publicBackendUrl = process.env.NEXT_PUBLIC_BACKEND_URL || "";
const BASE_URL = publicBackendUrl.replace(/\/$/, "");

function buildUrl(path) {
  return `${BASE_URL}${path}`;
}

export async function fetchRecipes() {
  const response = await fetch(buildUrl("/api/recipes"));
  if (!response.ok) throw new Error("Failed to load recipes");
  return response.json();
}

export async function createRecipe(recipeData) {
  const response = await fetch(buildUrl("/api/recipes"), {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(recipeData),
  });
  if (!response.ok) {
    const data = await response.json().catch(() => ({}));
    throw new Error(data.error || `Server error ${response.status}`);
  }
  return response.json();
}

The buildUrl helper keeps every fetch call consistent. All routing logic is in one place — if the base URL ever changes, only this file needs updating.

Part B — The Proxy Rewrite in next.config.mjs

Next.js rewrites /api/* and /auth/* requests to the Flask backend. The target URL is read from the BACKEND_URL environment variable, which is a server-side variable (no NEXT_PUBLIC_ prefix) because the rewrite only runs on the Next.js server, never in the browser.

Two different environment variables — easy to confuse:
  • BACKEND_URL — used by the Next.js server process to know where to forward proxy requests. It is never sent to the browser. This is the Docker service name URL: http://web:5000.
  • NEXT_PUBLIC_BACKEND_URL — if set, prefixes every fetch call made from browser-side code. For our setup this stays empty; the browser always calls /api/... on the same origin and the Next.js rewrite handles the rest.
// next.config.mjs

// Locally the Next.js dev server proxies to localhost:5001.
// Inside Docker Compose the backend is reachable via the service name "web".
const BACKEND_URL = process.env.BACKEND_URL || "http://localhost:5001";

const nextConfig = {
  // Required for the multi-stage Docker build — produces .next/standalone.
  output: "standalone",

  async rewrites() {
    return [
      { source: "/api/:path*",  destination: `${BACKEND_URL}/api/:path*` },
      { source: "/auth/:path*", destination: `${BACKEND_URL}/auth/:path*` },
    ];
  },
};

export default nextConfig;
Why output: "standalone"? Next.js compiles everything needed to run node server.js into a single .next/standalone folder. That folder is self-contained — no separate node_modules install is needed at runtime. This is what makes the final Docker image lean.

Part D — The Frontend Dockerfile (multi-stage)

Create recipes-frontend/Dockerfile. The build is split into two stages so the final image is as small as possible.

Notice that BACKEND_URL is declared as an ARG and immediately promoted to an ENV in both stages:

  • In the builder stage it is available when npm run build runs, so Next.js can bake the correct proxy target into the compiled server bundle.
  • In the runner stage it is available at runtime so the node server.js process can still read it if needed.
# ── Stage 1: builder ──────────────────────────────────────────────────────────
FROM node:20-alpine AS builder

WORKDIR /app

# BACKEND_URL is read by next.config.mjs during the build (for rewrites).
ARG BACKEND_URL=http://web:5000
ENV BACKEND_URL=$BACKEND_URL

# Install dependencies first — this layer is cached unless package.json changes.
COPY package.json package-lock.json* ./
RUN npm ci

COPY . .

RUN npm run build

# ── Stage 2: runner ───────────────────────────────────────────────────────────
FROM node:20-alpine AS runner

WORKDIR /app

ENV NODE_ENV=production
ARG BACKEND_URL=http://web:5000
ENV BACKEND_URL=$BACKEND_URL

# Copy only the standalone output and static assets from the builder stage.
COPY --from=builder /app/public            ./public
COPY --from=builder /app/.next/standalone  ./
COPY --from=builder /app/.next/static      ./.next/static

EXPOSE 3000

CMD ["node", "server.js"]
StageWhat Happens
builder Installs all npm dependencies and runs npm run build. BACKEND_URL is available here so the rewrite config is compiled with the correct target. Produces .next/standalone.
runner Starts from a fresh minimal Node image. Copies only the compiled output — no source code or dev dependencies. BACKEND_URL is also available here for runtime reads.

Part E — Build the Frontend Image Separately

Because docker-compose.yml lives in the backend repository, not the frontend repository, we cannot use build: with a relative path to the frontend source. Instead we build the image by hand first and give it a name that Compose can reference.

From inside the recipes-frontend directory, run:

docker build \
  --build-arg BACKEND_URL=http://web:5000 \
  -t recipeshare-frontend \
  .
FlagWhat It Does
--build-arg BACKEND_URL=http://web:5000 Passes the value into the Dockerfile ARG. The Next.js build uses it to set the proxy destination in next.config.mjs. web is the Docker Compose service name for the Flask backend.
-t recipeshare-frontend Tags the resulting image as recipeshare-frontend. This is the name Compose will reference.
. The build context — the current directory (the frontend source).
Rebuild when you change the code. Unlike docker compose up --build, a manually built image is not automatically refreshed when source files change. Re-run the docker build command whenever you want to push updated frontend code into the image.

Part F — Add the Frontend Service to docker-compose.yml

docker-compose.yml lives in the backend repository. The frontend service uses image: instead of build: because the image was already built in the previous step.

services:

  # ── Flask backend ───────────────────────────────────────────────────────────
  web:
    build: .                      # Dockerfile is in the same folder as this file
    ports:
      - "5001:5000"               # host:container — access on localhost:5001
    environment:
      FLASK_APP: app
      FLASK_ENV: development
      DATABASE_URL: postgresql://postgres:postgres@db:5432/recipeshare
      SECRET_KEY: dev-secret-change-in-prod
    depends_on:
      - db
    restart: unless-stopped

  # ── Next.js frontend ────────────────────────────────────────────────────────
  frontend:
    image: recipeshare-frontend   # pre-built image — run `docker build` first
    ports:
      - "3000:3000"
    environment:
      # Overrides the value baked in at build time if needed at runtime.
      BACKEND_URL: http://web:5000
      NODE_ENV: production
    depends_on:
      - web
    restart: unless-stopped

  # ── PostgreSQL database ─────────────────────────────────────────────────────
  db:
    image: postgres:16
    environment:
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
      POSTGRES_DB: recipeshare
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    restart: unless-stopped

volumes:
  postgres_data:
Port mapping — 5001:5000: Flask listens on port 5000 inside its container. We expose it as port 5001 on the host so you can test the API directly from your laptop (http://localhost:5001/api/recipes). Inside the Docker network every container talks to Flask on the internal port 5000 — so the frontend uses http://web:5000, not 5001.

Part G — Start the Full Stack

From the backend repository folder (where docker-compose.yml lives):

# 1. Start all services
docker compose up

# 2. In a separate terminal — run migrations once on first start
docker compose exec web flask db upgrade

Then open:

  • Frontend: http://localhost:3000
  • Backend API (direct): http://localhost:5001/api/recipes

Part H — How a Request Flows End-to-End

Here is what happens when a user visits http://localhost:3000 and the page fetches recipes:

1
The browser calls GET /api/recipes. Because the page was served from port 3000, this request goes to the Next.js server — not to Flask.
2
buildUrl("/api/recipes") in api.js returns /api/recipes (empty base URL in the browser context). Next.js matches the rewrite rule and forwards the request to http://web:5000/api/recipes on the internal Docker network. The browser never speaks to Flask directly.
3
Flask receives the request, queries PostgreSQL at hostname db, and returns JSON.
4
Next.js relays the JSON response back to the browser.
Why no CORS errors? The browser only ever talks to one origin — localhost:3000. The Next.js rewrite makes the frontend and backend appear to be the same server from the browser's point of view. Flask never receives a cross-origin request directly from the browser.

Frontend build succeeds but the app crashes at startup with "cannot find server.js"

The most common cause is that output: "standalone" is missing from next.config.mjs. Without it, Next.js does not produce the .next/standalone folder and server.js does not exist. Add the option, then rebuild the image:

docker build --build-arg BACKEND_URL=http://web:5000 -t recipeshare-frontend .
API calls return 502 or "connection refused"

The frontend container cannot reach the backend. Check:

  • The image was built with --build-arg BACKEND_URL=http://web:5000 (service name web, internal port 5000 — not 5001).
  • The BACKEND_URL env var in docker-compose.yml also says http://web:5000.
  • The web service started successfully: docker compose logs web.
I updated frontend code but the running container still shows old content

The image is not rebuilt automatically. You must run docker build again from the recipes-frontend directory, then restart the Compose stack:

docker build --build-arg BACKEND_URL=http://web:5000 -t recipeshare-frontend .
docker compose up
